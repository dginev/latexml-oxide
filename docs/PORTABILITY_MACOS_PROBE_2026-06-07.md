# macOS portability probe — findings (2026-06-07)

Point-in-time diagnostic for issue
[#217](https://github.com/dginev/latexml-oxide/issues/217) (Portable use on
MacOS), Phase 0 of the portability ladder in
[`RELEASE_CRITERIA.md`](RELEASE_CRITERIA.md) §3. Facts gathered by
`tools/portability_probe.sh` running in `.github/workflows/portability.yml`
on `macos-15` (arm64, 3 vCPU / 7 GB) runners, under two TeX providers:
**BasicTeX cask** (MacTeX's small distribution — what a typical Mac TeX user
has) and **Homebrew `texlive` formula** (built from TL source with
`--enable-shared`).

Runs: baseline `27092132824` (probe sections; builds invalidated by a
harness `--locked` bug), corrected `27092295937` (build + smoke results).
Reports are uploaded as `probe-report-{basictex,brew-texlive}` artifacts.

## Headline verdicts

1. **MacTeX/BasicTeX ships ZERO kpathsea dev artifacts.** No `kpathsea.h`,
   no `libkpathsea*`, no `kpathsea.pc` — anywhere (TL tree `/usr/local/
   texlive/2026basic`, `/Library/TeX`, brew prefix, system paths). Only the
   `kpsewhich` binary ships (kpathsea 6.4.2). The #217 hope that "the mactex
   installation may already bring the relevant headers" is **dead**: linking
   `libkpathsea` on a MacTeX-only Mac is *impossible*, not misconfigured.
   (Same on Linux: a vanilla TL 2025 tree carries no dev artifacts either —
   Debian's come from the separate `libkpathsea-dev` package.)

2. **Homebrew `texlive` ships the complete kpathsea dev stack.**
   `kpathsea.pc` is symlinked into `/opt/homebrew/lib/pkgconfig` (on
   pkg-config's default search path — the formula is not keg-only), next to
   `libkpathsea.dylib` and full headers. `pkg-config --exists kpathsea`
   succeeds out of the box → `kpathsea_sys` builds **unchanged** on a
   brew-texlive Mac.

3. **libxml2 / libxslt / libexslt all resolve via pkg-config** once the
   keg-only paths are exported:
   ```sh
   export PKG_CONFIG_PATH="$(brew --prefix libxml2)/lib/pkgconfig:$(brew --prefix libxslt)/lib/pkgconfig"
   ```
   This is the exact incantation the README's macOS section must document.

4. **`latexml_post/build.rs` was redundant — deleted** (commit
   `b8b561dbed`). Its bare `cargo:rustc-link-lib=xslt`/`exslt` lines (no
   search path — the macOS-fragile part) duplicated what the `libxslt`
   0.1.3 crate's own build script already does better: pkg-config probe
   with a bare-link fallback. Verified on Linux that the binary still
   links `libxslt.so.1`/`libexslt.so.0` and converts correctly without it.

5. **Toolchain prerequisites are all present on macOS runners**: Apple
   clang 17, GNU make 3.81, `libclang.dylib` (Xcode toolchain + CLT) for
   bindgen, `/bin/sh` for libmarpa's pre-generated `configure` (autoconf
   itself is absent but not needed — the vendored `libmarpa-8.6.2.tar.gz`
   is a dist tarball). Rust nightly installs cleanly via
   `dtolnay/rust-toolchain@nightly`.

6. **The runners are TL 2026** (`/usr/local/texlive/2026basic`, brew
   `texlive 20260301`) — we embed only TL2023/TL2025 dumps. Year detection
   survives both odd tree shapes via the `pdflatex --version` fallback
   ("TeX Live 2026" / "TeX Live 2026/Homebrew"), since
   `SELFAUTOPARENT` parses fail (`2026basic` is not a bare year;
   `/opt/homebrew/Cellar/texlive` has no year). The resolver then falls
   back to the most-recent embedded year (2025) per
   `dump_paths.rs` — dump-over-mismatched-tree is exercised by the smoke
   runs below. Follow-ups: (a) generate `latex.2026.dump.txt` via
   `tools/make_formats.sh` on a TL2026 host; (b) minor robustness —
   `parse_year_str` could accept leading digits so `2026basic` parses
   without needing `pdflatex` on PATH.

## Build + smoke results (run 2, corrected harness)

Run `27092295937`. Crate-by-crate native-binding attribution:

| binding | basictex | brew-texlive |
|---|---|---|
| `libxml` (libxml2 + bindgen) | **OK** | **OK** |
| `libxslt` | **OK** | **OK** |
| `kpathsea` | FAIL — `kpathsea_sys` build.rs `panic!`: "Could not find kpathsea using pkg-config" (correct: nothing exists to find) | **OK** |
| `marpa` (vendored libmarpa `./configure && make` + bindgen) | **OK** | **OK** |
| workspace (`--profile ci`) | FAIL at `kpathsea_sys` (same panic) | FAIL at **final binary link** — see below |

**The brew-texlive leg compiled the entire workspace** and failed on
exactly **one undefined symbol** at the `latexml_oxide` link:
`xsltMaxDepth` (no leading underscore). Root cause is a **new class of
portability bug — pregenerated bindgen bindings are platform-locked**:
the `libxslt` 0.1.3 crate ships `bindings.rs` generated by bindgen *on
Linux*, where statics get `#[link_name = "\u{1}xsltMaxDepth"]` — the
`\u{1}` escape pins the raw **ELF** symbol name and bypasses Mach-O's
`_`-prefix decoration. The brew dylib **does export** `_xsltMaxDepth`
(verified by `llvm-nm` on the arm64 bottle); the linker was simply told
to look for the wrong name. Functions don't carry `link_name`, which is
why all other libxslt symbols linked. 12 statics in the crate's
bindings are affected; latexml-oxide referenced only this one
(`latexml_post::xslt::set_xslt_max_depth`, the Perl
`XML::LibXSLT->max_depth(1000)` parity knob).

**Fix landed on this branch**: `set_xslt_max_depth` now resolves the
global via `libc::dlsym(RTLD_DEFAULT, "xsltMaxDepth")` — `dlsym`
applies the platform's own C-symbol decoration, so the same code works
on ELF and Mach-O, preserving the Perl cap=1000 parity on both. Unit
test `max_depth_tests::dlsym_sets_perl_parity_cap` reads the value back
through the same path. Upstream follow-up for rust-libxslt: drop the
`\u{1}` link_names from statics (plain `extern "C"` statics let Rust
apply per-platform decoration), or generate bindings at build time.

**Run 3 (`27092594389`, after the dlsym fix): brew-texlive leg fully
GREEN** — workspace build OK, `latexmlmath '1+1=2'` OK, hello.tex
conversion OK. That smoke run also exercises the dump-year fallback in
anger: a TL2026 tree served by the embedded TL2025 dump. **macOS arm64
is a working build host today** via `brew install texlive libxml2
libxslt` + the `PKG_CONFIG_PATH` export. The basictex leg stays red at
the `kpathsea_sys` panic until the Phase 1 subprocess-`kpsewhich`
fallback exists (decided 2026-06-07: implemented **upstream in
rust-kpathsea first**, then dep-bumped here) — its red status is honest
signal, not noise.

## The kpathsea dichotomy → Phase 1 plan

Build-host story (CI, source builds):

| TeX provider | libkpathsea linkable? | What's needed |
|---|---|---|
| Homebrew `texlive` | **yes, today** | nothing — pkg-config finds it |
| BasicTeX / MacTeX | **no — nothing to link** | subprocess fallback (below) |
| Debian/Ubuntu | yes (`libkpathsea-dev`) | current state |

The Perl ground truth is decisive here:
`LaTeXML/lib/LaTeXML/Util/Pathname.pm` **never links libkpathsea** — it
shells out to the `kpsewhich` executable (`$LATEXML_KPSEWHICH` override,
L395) and builds its own `ls-R`-based `$kpse_cache` (L396-426). So a
**subprocess-`kpsewhich` fallback is the most Perl-faithful option**, and
the only one that can serve MacTeX-only users — both as build hosts and as
*consumers of prebuilt binaries* (a distributed mac binary that dynamically
links `libkpathsea.dylib` would fail to even load without brew texlive;
see `RELEASE_CRITERIA.md` §11's self-contained bar).

The Rust-side surface is ideally small: the entire in-process kpathsea
dependency funnels through `latexml_core/src/util/pathname.rs` —
a `KPSE: Lazy<Mutex<Option<Kpaths>>>` singleton, the perf-only
`prewarm_kpathsea()`, and `kpsewhich(candidates)` with **one** call site
(`binding/content.rs:2634`).

**IMPLEMENTED upstream 2026-06-07** (decision: subprocess + ls-R cache,
in rust-kpathsea first; the c2rust-transpile alternative was rejected —
kpathsea is LGPL, so an embedded transpiled copy would encumber every
shipped artifact and end the §4 public-domain ambition, besides freezing
a snapshot of a library whose entire value is staying in sync with the
host). rust-kpathsea branch `subprocess-fallback`, kpathsea 0.3.0 /
kpathsea_sys 0.2.0:

1. `kpathsea_sys` build.rs no longer panics: `KPATHSEA_LIB_DIR` override
   → pkg-config → graceful unlinked build, publishing
   `DEP_KPATHSEA_LINKED` via the `links = "kpathsea"` channel.
2. `Kpaths` is backend-dispatched at compile time: in-process FFI when
   linked (unchanged fast path); otherwise subprocess-`kpsewhich`
   fronted by a one-shot ls-R cache — a faithful port of Perl
   `pathname_kpsewhich` + `build_kpse_cache`, with two deliberate
   divergences (first-wins across TEXMF trees instead of Perl's
   last-wins overwrite; unconditional `-dev` subdir skip, required for
   first-wins correctness). New API: `new_subprocess()`,
   `with_kpsewhich(path)`, `find_first(&[..])`, `is_in_process()`;
   `KPSEWHICH` env override.
3. Bonus fix while in there: the `guess_format_from_filename`
   alt-suffix overflow panic (the bug oxide guards with `catch_unwind`
   in `pathname.rs`) now carries the missing length guard upstream.
4. oxide consumes it via `[patch.crates-io]` (git branch) until the
   crates.io release; `prewarm_kpathsea()` returns early on the
   subprocess backend (no format table to warm; ls-R cache builds
   lazily instead).
5. **macOS distribution artifact** follow-up: build with subprocess-only
   kpathsea → no `libkpathsea.dylib` in `otool -L` output, runnable on
   any Mac with any TeX distribution on PATH.

Validated in both configurations on Linux (linked, and unlinked via a
pkg-config view hiding only `kpathsea.pc` — the MacTeX simulation);
the basictex CI leg is the macOS ground truth.

## Remaining macOS ladder items (beyond kpathsea)

- README macOS section: `brew install libxml2 libxslt` (+ `texlive` for
  source builds, or BasicTeX/MacTeX for runtime), the `PKG_CONFIG_PATH`
  export, nightly toolchain note.
- CI: promote the probe to a gating macOS job once green — build + a
  TeX-light test tier first (full 1334-test suite needs tlmgr package
  fills on BasicTeX).
- `ubuntu-24.04-arm` job — RELEASE_CRITERIA §3 stage 2 (aarch64 Linux)
  is nearly free now and covers the other arm64 leg.
- `rust-toolchain.toml` nightly pin (also #143's reproducibility ask) so
  all matrix legs compile the same compiler.
- `Cargo.lock` is currently **gitignored** (`.gitignore:13 *.lock`) — CI
  checkouts re-resolve dependencies on every run. Committing it is the
  usual practice for binary projects and would firm up cross-platform
  reproducibility (separate policy decision; flagged, not changed).

## Windows ledger (deferred, RELEASE_CRITERIA §3 stage 5)

Known structural blockers recorded for honesty of the ladder:
`libmarpa-sys` builds via `./configure && make` (needs a `cc`-crate
rewrite for MSVC; tractable — the tarball is vendored); `lsp_server`
uses unix sockets/`poll`; `graphics.rs`/`graphics_cache.rs` use
`cfg(unix)` paths (all POSIX-portable — fine for macOS, absent on
Windows); libxml2/libxslt via vcpkg; MiKTeX vs TL kpathsea differences —
the subprocess-`kpsewhich` mode removes the hardest one.
