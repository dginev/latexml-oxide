% Minimal reproduction of the expl3 l3file expansion chain issue.
%
% CONTEXT: expl3-code.tex line 11527-11528 uses \exp_last_unbraced:NNNNo
% to define \__file_name_expand_cleanup:w with a delimited parameter.
% The expansion chain works correctly in isolation but the definition
% is lost during expl3-code.tex loading.
%
% WHAT WORKS:
%   - The expansion chain \exp_last_unbraced:NNNNo correctly produces tokens
%   - When called in post-load context (raw_tex), the definition persists
%   - All other l3file functions defined via direct \cs_new:Npn work fine
%
% WHAT FAILS:
%   - During expl3-code.tex loading, \gdef for \__file_name_expand_cleanup:w
%     never reaches do_def() (confirmed by tracing)
%   - The \tex_gdef:D primitive IS invoked 2213 times during loading
%   - But the specific cleanup:w definition is NOT among them
%   - The \exp_last_unbraced:NNNNo expansion at line 11527 produces tokens
%     that go through the expandafter chain → \cs_new:Npn → \cs_gset:Npn →
%     \tex_long:D \tex_gdef:D, but the \gdef body closure is never called
%     for the cleanup:w CS
%
% SUSPECTED ROOT CAUSE:
%   The invoke_token Expandable case calls read_x_token(fully_expand=true).
%   This expands the entire chain INSIDE read_x_token:
%     \exp_last_unbraced:NNNNo → expandafter chain → \cs_new:Npn expansion →
%     \cs_gset:Npn expansion → \tex_long:D \tex_gdef:D
%   The \tex_long:D is returned (not expandable). The invoke_token loop
%   processes it and breaks. The outer load_tex_definitions loop should then
%   pick up \tex_gdef:D. But somehow the definition is lost.
%

%   4. Verify the pushback contains the right tokens after the expansion
