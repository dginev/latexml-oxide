1//! delarray.sty — array delimiter package.
2//!
3//! delarray.sty (D. Carlisle, 1991-1994) provides the
4//! `\begin{array}{l}\left(...\right)` delimiter-augmented form. It
5//! `\RequirePackage{array}` then `\def\@@array[#1]{...}` to wrap the
6//! kernel `\@array` with delimiter detection.
7//!
8//! Perl LaTeXML has no `delarray.sty.ltxml`. With its default
9//! `INCLUDE_STYLES=false`, the raw `delarray.sty` is NOT loaded —
10//! Perl emits a "missing binding" warning and continues with the
11//! binding-aware `\@@array` from `latex_constructs.rs`. The user's
12//! `\begin{array}{l}\left(\right)` then digests cleanly because our
13//! `\@@array` binding plus inline `\left\right` already cover this
14//! pattern.
15//!
16//! In Rust we default to `INCLUDE_STYLES=true` (ar5iv preload sets
17//! it). Without this stub the raw `delarray.sty` IS loaded — its
18//! `\def\@@array[#1]{...}` overwrites our binding-aware `\@@array`
19//! and the next `\begin{array}` falls through to the LaTeX kernel
20//! `\@array`, which needs `\@classz`/`\@acol`/... (not defined by our
21//! `array.sty.ltxml` stub). Result: cascade of "undefined" errors and
22//! a fatal TooManyErrors abort.
23//!
24//! Stubbing this binding suppresses the raw-load (Perl-parity) so
25//! `\@@array` keeps pointing at our binding. The delimiter detection
26//! in the user's source still works through the standard array+`\left
27//! \right` path.
28//!
29//! Witnesses: canvas-3 0809.4328, 0810.2088, 0810.2091, 0811.2514,
30//! 0811.4484, 0812.1967, 0901.2107, 0901.3167.
31use crate::prelude::*;
32
33LoadDefinitions!({
34  RequirePackage!("array");
35});
